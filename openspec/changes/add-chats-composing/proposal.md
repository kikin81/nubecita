## Why

The Chats tab ships today as a **read-only** DM viewer (epic `nubecita-nn3`): users can browse the conversation list and read threads, but there is no way to **reply in a conversation** or **start a new one**. The tab surfaces incoming DMs yet can't participate in them. This change makes Chats a working 1:1 text DM client, implementing epic `nubecita-b6uv` (Chats V2 — composing + sending, text only). The platform is already in place: `chat.bsky.convo.sendMessage` is generated by atproto-kotlin, the `transition:chat.bsky` scope + `atproto-proxy` header are configured, and `resolveConvo` (over `getConvoForMembers`) already resolves/creates a 1:1 convo from a peer DID — so V1 is purely client work.

## What Changes

- **Send in an existing convo:** add a composer (M3 Expressive text field + end-aligned send button, disabled while blank, IME `Send`) to the chat thread screen; sending appends the message **optimistically** with a *sending* indicator, reconciles to *sent* on server confirm, and shows an inline **retry** on failure.
- **Repository sending:** add `sendMessage(convoId, text)` to `ChatRepository` / `DefaultChatRepository` over `chat.bsky.convo.sendMessage`, with failure mapping consistent with the existing `toChatError`.
- **Start a new conversation:** add a New-Chat FAB (M3 Expressive small FAB, bottom-end) to the conversation-list screen that opens a **recipient-picker** screen (handle search via `app.bsky.actor.searchActors`); selecting a result navigates into the thread for that `otherUserDid`, which resolves-or-creates the convo via the existing `resolveConvo` path.
- **MVI surface growth:** extend the thread's `ChatScreenViewState` / `ChatEvent` / `ChatEffect` to carry composer text, a per-message **send status** modeled as a sealed sum (not `Async<T>`), and `Send` / `RetrySend` events plus a transient error effect; add a recipient-picker screen with its own VM + contract.
- **Navigation:** add a `NewChat` nav key in `:feature:chats:api`; tab-internal pushes go through `UiEffect` + `LocalMainShellNavState` (ViewModels never inject nav state).
- **Tests & previews:** unit tests for send / optimistic echo / retry and the recipient-picker VM; screenshot tests for composer states (empty, typed, sending, failed), the FAB, and the picker; extend `FakeChatRepository`.

Out of scope (deferred): media attachments, reactions, typing indicators, read receipts, message edit/delete, group DMs, new-message push notifications, in-conversation search, GIFs/stickers/voice, and real-time streaming (periodic refresh suffices for V1).

## Impact

- Affected specs: `feature-chats` (new capability spec — ADDED requirements for composing, optimistic delivery, send retry, and new-conversation creation).
- Affected code:
  - `:feature:chats:api` — new `NewChat` nav key.
  - `:feature:chats:impl` — `ChatRepository` / `DefaultChatRepository` (`sendMessage`); `ChatContract` / `ChatViewModel` / `ChatScreen` (composer + optimistic send); `ChatsScreen` (New-Chat FAB); new recipient-picker screen + VM + contract; `ChatsNavigationModule` (register entry); `ChatsErrorMapping` (send errors); `FakeChatRepository` (test double); `MessageUi` / `ThreadItem` (send-status field).
- No platform/auth/build impact: no new OAuth scope, no client-metadata change, no atproto-kotlin upgrade.
