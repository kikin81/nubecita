## 2. Chats / Requests segmentation — G1 (`nubecita-kc17.2`) (D-3, D-4, D-9)

- [ ] 2.1 Extend `ChatRepository` / `DefaultChatRepository` to hold both accepted and request lists. Decide and document the interface shape for exposing requests — either a `status`-parameterized `observeConvos(segment)` or a sibling `observeRequestConvos()` (the existing `observeConvos()` is the starting point). `refreshConvos()` issues `listConvos(status="accepted")` + `listConvos(status="request")` **concurrently** (`async` within a `coroutineScope`, they're independent — one round-trip of latency), each populating its own cached list; a request-fetch failure leaves the accepted list intact. Repo unit tests: two-list refresh, concurrent dispatch, partial failure keeps accepted.
- [ ] 2.2 `ChatsContract`: add `activeSegment: ChatsSegment` and `requestCount: Int`; add `ChatsEvent.SegmentSelected`. `ChatsViewModel` projects the active segment's list and the request count. VM unit tests: default segment Chats, switch to Requests, request-count derivation.

A Requests-fetch failure renders an inline error body in the Requests segment only. Screenshot tests: Chats segment, Requests segment with badge, Requests inline error, light/dark.

## 4. Leave-with-undo — G3 (`nubecita-kc17.4`) (D-7, D-8)

- [ ] 4.1 ViewModel-owned deferred leave: hold a single pending-leave batch (set of convoIds); on `LeaveSelected`, remove the rows from the projected list immediately and emit `ShowLeaveUndo(count)`; on snackbar dismiss commit (call `leaveConvo` once per held convo); on `UndoLeave` restore with no network. Starting a new leave commits the prior batch first. Launch the terminal commit on an **injected application-scoped `CoroutineScope`** (not `viewModelScope`) so it survives the VM tearing down mid-call. The optimistic projection is applied over the freshest cache so a concurrent refresh can't resurrect a mid-undo row. **Partial failure is per-conversation:** when a flushed batch's `leaveConvo` calls partly fail, restore only the failed conversations and keep the rest removed. VM unit tests: defer (no call while pending), commit-on-dismiss, restore-on-undo, supersede-prior-batch, bulk, and bulk partial-failure restores only the failed convos.
- [ ] 4.2 `onCleared()` flush: when the Chats VM is cleared normally (navigated away), commit any pending leave through the application scope rather than dropping it (leaving a chat then navigating away must leave it). Process death — where `onCleared()` does not run — drops the pending leave (no persistence). VM/unit test: `onCleared()` with a pending batch issues the `leaveConvo` calls on the app scope.
- [ ] 4.3 `ChatsScreenContent`: drive the leave-undo Snackbar from `ShowLeaveUndo` via the existing `SnackbarHostState`; Undo action sends `UndoLeave`. Screenshot/preview of the post-leave list + snackbar where feasible (else covered by VM tests).
