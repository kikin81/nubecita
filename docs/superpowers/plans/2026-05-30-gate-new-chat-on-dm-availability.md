# Gate New-Chat picker + FAB on DM availability — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop the New-Chat recipient picker from listing actors the user can't DM, and hide the New-Chat FAB when the account isn't DM-enrolled.

**Architecture:** Promote the existing `canViewerMessage` gate from `:feature:profile:impl` into a shared `:core:profile` helper over the common `(ProfileAssociated?, ViewerState?)` shape. Thread a fail-open `canMessage` boolean through `ActorUi`, the two `:core:actors` search mappers, and an additive Room v3→v4 cache column. Filter `!canMessage` in `NewChatViewModel` (recent + search). Hide the FAB on `InitialError(NotEnrolled)`. The existing `ChatsErrorMapping` "can't send messages" screen stays the authoritative post-tap backstop for cache staleness.

**Tech Stack:** Kotlin 2.3, Hilt, Room (AutoMigration), Jetpack Compose, JUnit Jupiter + MockK + Turbine (unit), AndroidX Room/Compose screenshot tests (instrumented).

**bd issue:** nubecita-j1gm · **branch:** `fix/nubecita-j1gm-gate-new-chat-on-dm-availability` · **spec:** `docs/superpowers/specs/2026-05-30-gate-new-chat-on-dm-availability-design.md`

```kotlin
package net.kikin.nubecita.core.profile

import io.github.kikin81.atproto.app.bsky.actor.ProfileAssociated
import io.github.kikin81.atproto.app.bsky.actor.ViewerState

/**
 * Resolves whether the signed-in viewer is allowed to send this actor a DM,
 * from the common `(associated, viewer)` shape carried by every `ProfileView*`.
 *
 * Reads `associated.chat.allowIncoming`:
 * - `"none"` → nobody accepts DMs → false.
 * - `"following"` → only accounts the actor follows; encoded viewer-side as
 *   `viewer.followedBy != null` (the actor's follow record points at the viewer).
 * - `"all"` / absent `associated.chat` → true (fail-open, matches official clients).
 *
 * This is a *hint* derived from the search/profile response, not an authoritative
 * check; the post-tap `getConvoForMembers` error remains the source of truth.
 */
fun canViewerMessage(
    associated: ProfileAssociated?,
    viewer: ViewerState?,
): Boolean =
    when (associated?.chat?.allowIncoming) {
        ALLOW_INCOMING_NONE -> false
        ALLOW_INCOMING_FOLLOWING -> viewer?.followedBy != null
        else -> true
    }

private const val ALLOW_INCOMING_NONE = "none"
private const val ALLOW_INCOMING_FOLLOWING = "following"
```

```kotlin
@Entity(tableName = "actors")
data class ActorEntity(
    @PrimaryKey val did: String,
    val handle: String,
    @ColumnInfo(name = "display_name") val displayName: String?,
    @ColumnInfo(name = "avatar_url") val avatarUrl: String?,

fun ActorEntity.asExternalModel(): ActorUi =

fun ActorUi.toCacheEntity(lastSeenAt: Instant): ActorEntity =
    ActorEntity(did = did, handle = handle, displayName = displayName, avatarUrl = avatarUrl, lastSeenAt = lastSeenAt, canMessage = canMessage)
```
